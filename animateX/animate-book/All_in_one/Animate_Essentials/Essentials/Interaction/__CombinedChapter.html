Adding Interactivity

Making compositions interactive

Get ready for Action

This chapter will lead you the way into adding interactivity to your compositions. You will learn how to add actions, how to listen and react to events and how to use the code wizzard. Don't be affraid, it won't hurt. I promise!

A few remarks on the sideline

Edge Animate is not only a tool for creating motion and animations, but creating interactive content as well. In order to do that, you can use Actions and the Edge Animate API. The underlying language here is JavaScript. Fortunately, you don't have to be able to write code yourself (although it might in fact prove to be helpful). Edge Animate comes with a code wizzard, that provides you with predefined code snippets you can use for adding interactivity. We will show you how this is done later in this chapter.

These code snippets can be inserted at several points in Edge Animate. In order to give you a little help, the interface of Edge Animate shows these curly braces "{}" whenever you can enter actions.

When starting to work with Edge Animate, the provided code snippets should cover pretty much everything you need. Nevertheless, it might be a good idea to practice writing short scripts yourself and get used to the language. You might want to take a look at the Cheat Sheet we have included here, which features the most important actions and events. Also stay tuned in for the advanced addition, in which we will tackle a lot more possibilities of working with code. Also make sure to check out the recommended sources in the introductory chapter, they will also help you with getting to know more JavaScript.

If you are already fit at using JavaScript, you can bring in all your knowledge here and build rich applications with Edge Animate.
The sky is the limit.

Adding Interactivity

Listening to Events

Wait for it...

In this section you will learn about events and how to respond to them in order to introduce interactivity. You will get to know the different types of events you may encounter in the Edge Animate context and learn about their specifics.

Introduction

An important aspect of interactivity is the ability to listen and respond to Events. Whenever a certain event occurs, corresponding code is being executed. In Edge Animate, such code snippets are called Actions. Events can be triggered for several reasons and by different parties. An obvious example is the click event, that is triggered when the user clicks on an element. Other events can be triggered by the composition itself, the superior system or through trigger on the Timeline. Here is a quick overview of relevant event types you may encounter in Edge Animate and which will be discussed in detail further below:

  • User Events
  • Timeline Events
  • Composition Events
  • Symbol and Element Events

Handling Events

Now that we have an idea of what events are, let's take a look at what we can do and how we can respond to them. We have already learned that Edge Animate indicates the option to respond to events with curly braces. When you click on these braces, the Actions window opens up. Note that this window always relates to the context and/or the currently selected element. In this regard, the stage is an element on its own. Depending on the context and element, you have different events you can respond to. By selecting one of the given options, you can add an action to this event.

You can add actions to several events for the same element. Use the "+" and "-" buttons to add and remove events. Each event will be displayed as a tab. Once you've selected an event, you can define the actions in the code panel by writing your own custom script or with the help of the code wizzard. Note that all the actions relating to the same event are being placed in the same tab in the code panel.

The Actions you define in this code view have to be valid JavaScript. If you are not familiar with this language, you should use the provided scripts we will explain in more detail in the following Actions section.

For now let's first take a closer look at the different kind of events and see what they have to offer.

User Events

User Events play an important role in terms of interactivity. These events are triggered by the user with a mouse, keyboard or through touchscreen. Most of these events are available for every element as well as symbols and the stage. There are a few exceptions, though, that relate only to the composition as such and are therefore only available for the stage.

But first we will look at the user events that can be applied everywhere. The first group of user events are mouse events:

click

Click events are triggered as soon as the user clicks on the selected element.

dblclick

Similar to the click event, only this time the event is triggered by a double click on the element.

mouseover

This event is triggered when the mouse cursor touches the element. This effect is also known as "hover". Be careful, though, since this event is triggered every time the cursor touches the element, which means it will be triggered again when passing through a nested element within. This could lead to unintended behavior, so in those cases it is better to use mouseenter.

mouseout

As opposite of mouseover, this event is triggered when the mouse cursor leaves the element. Since the same issues may occur as with mouseover, it is better to use mouseleave instead.

mousedown and mouseup

Sometimes a click event as a whole is not sufficiant. This might be the case when you want to react to the pressing and releasing of the mouse button seperately. Then you can use mousedown for this first phase of a click, and accordingly mouseup for releasing the mouse button. The actual click event is triggered after mouseup.

mousemove

When applied, this event is triggered permanently as long as the mouse cursor moves over the particular element. This can be useful for example when you want to respond to the current position of the mouse cursor.

mouseenter

As mentioned above, mouseenter is similar to mouseover, only that in this case, the event is only triggered again when the cursor actually left the boundaries of the element. This makes it usually a better choice for creating hover effects.

mouseleave

Correspondend to mouseenter is this event the alternative for mouseout. It is only triggered once when the cursor leaves the area of the element, ignoring child elements.

focus

The focus event is fired when an element has received focus. An element can gain focus via keyboard commands, such as the Tab key, or by mouse clicks on the element.

Here's an overview of available user events:

Apart from the abovementioned mouse events, there are also a couple of events specific for touchdevices. You have to be a bit careful here, when you want to apply mouse events and touch events at the same time. This might result in the action being executed twice. There are some workarounds, but they are a bit more advanced. These will be included in the advanced edition.

touchstart

As soon as the user touches the element on the screen, this event is triggered.

touchmove

This event is triggered permanently when the user moves the finger across the screen after a touchstart. It is useful to realize drag and drop features, for example.

touchend

Corresponding to touchstart, the touchend event is triggered when the user removes the finger from the surface.

swipeleft/swiperight

This event is triggered when a swipe gesture is used on the element, to the left or to the right side, respectively.


User Events on stage level

In the following part we will quickly walk through the user events on page level. As mentioned earlier, they can be accessed through the stage.

scroll

The scroll event is triggered when the user uses scrolling inside the composition.

keydown and keyup

This event is triggered whenever a key is pressed (down) or released (up). You can also listen to specific keys, but this is again a bit advanced and will be a topic in a different edition.

resize

When the size of the browser window is changed, this event is being triggered. Therefore it might be useful in the context of responsive compositions.

orientationChange

Specific to mobile devices, this event is triggered when the devices is turned. This allows your composition to adopt to landscape or portrait mode for example.

Timeline Events

Not only user interaction can trigger events. The timeline can trigger events as well, in addition to the triggers we have already encountered. These events relate to the position of the Playhead.

play

As you have probably guessed, this event is triggered when the Playhead starts playing, e.g. when it was stopped before.

stop

Accordingly, this event is being triggered when the Playhead is being stopped.

update

The update event is triggered the entire time the Playhead moves on that particular Timeline. Keep in mind, that this results in quite a lot of events being triggered, so it might affect your performance.

complete

The complete event is triggered once when the playback of that Timeline has been completed and the Playhead has reached the end of the Timeline.

Composition Events

There are two events that relate to the entire composition, compositionReady and onError. You can respond to these events through the stage. There are of course other events as well that can be applied to the stage, such as user events and symbol events.

Here we want to focus on the two composition events:

compositionReady

compositionReady is an event that gets triggered when the composition is initialized. Responding to this event allows you to insert code that is required for the further preparations of the composition. This would be the right place for the initial definition of variables or implementing the general application logic.

onError

This event relates to the entire page in which the composition is placed. Whenever an even on that page causes an error, the onError event is triggered. It therefore serves for error recovery.

Symbol events

In regards to symbols, you have several options for events. Regarded from the outside, a symbol is like an element (it in fact has a containing symbol element). Therefore you can apply all the above mentioned user events here as well and in the same way. Inside the symbol instance, you have the symbol timeline, where you can of course relate to the timeline events. In addition to that, there are two events that relate to the creation process and which can only be found with symbols and the stage: creationComplete and beforeDeletion. In order to listen to these events, you have to use the curly braces at the top of the Property Inspector on the symbol scope itself - or on the stage, respectively.

Let's have a look at these two options:

creationComplete

The creationComplete event gets triggered when the symbol gets instantiated, so whenever the symbol gets placed in the composition. Therefore this event allows you to implement instructions that shall be executed when the symbol get initialized, e.g. sending a notification, triggering other events or retrieving informations and setting variables.

beforeDeletion

This event on the other hand can be used to "clean up" before the symbol is finally removed from the stage.

Introducing Actions

Getting things done.

This section will give you an introduction into the topic of using actions in Edge Animate. You will learn about the different options, how to add and remove actions and which actions you will most frequently use.

Introduction

Actions are the code snippets that contain the information of how the element, symbol, timeline or composition should respond to a certain Event, which we have got to know a little better in the previous section. Actions are written in JavaScript and are in fact functions. If you are already familiar with the language, you can apply your entire skills here. But even if you don't have any clue, you can use Animate to add actions with ease, thanks to the code wizzard, which offers presets for code snippets. This way, you just have to make some adjustments to get the intended effect.


Where can I add actions?

You can add actions in the code view, which opens up when you click on the curly braces we have already seen in the last section. You can also right click on elements on the stage and select "Open Actions for...". Remember that these code views always relate to a specific context, for example an element or the stage. Another option is to use the general code window, which contains all events and actions of your composition. This general code window is definitely the better option when you deal with more complex interactions, since you have everything at hand, but at the same time it is more complicated for beginners. So it is rather recommended for advanced users. To open this code window go through Window > Code in the main menu bar.

With the navigation bar on the right hand side you can jump to every location where actions are applied, which makes it easier to keep an overview and to find specific actions in your composition. Be careful, though, as you can accidentaly create errors by adding code at the wrong place and crash your entire composition. Also, the code wizzard is not available in the code window. And we would suggest to stick to using this until you are fit enough using JavaScript.

We will therefore focus on using the code wizzard for now. If you prefer typing in actions manually, you should check out the Cheat sheet with the most frequently used actions we have provided. This will also give you a good overview and short descriptions to get you started right away.

Using the code wizzard

The code wizzard is a helping feature for adding actions to events. It offers you predefined code snippets for different purposes, that can easily be adopted to create valid actions. You simply pick an action and where necessary a target to add the appropriate code for the current event.

Tip: In case you are not going to use the code wizzard, you can collide and expand the panel with button underneath the code view, giving you a better view on the code.


How to work with the code wizzard

First of all, keep in mind that the code view above the wizzard is in fact an editor, which allows you to type in and edit the code. Using the code wizzard simply inserts presets into the editor, which can then be modified for the task at hand.

In order to insert such a preset into the code editor, first select the appropriate action with the selection on the left hand side. You can use the search field if you already have a specific action in mind or rifle through the menu. Selecting one of the options will reveal the available presets for each case. For example Links let's you choose between Link in Same Window and Link in New Window.

As soon as you click on one of the options in the right column, the equivalent code preset is inserted in the code editor above, along with comments that help you understand the code and explain possible modifications. Comments are displayed in green letters and follow a double slash: "//". These lines are not being executed as code. They are only there to make the code easier to understand for other human beings.

Controlling the timeline

Adding a link is a pretty simple use case. More often, you want to introduce actions in order to control the playback of the various animations of your composition. There are presets for those cases in the code wizzard as well. The first group of presets is called Playback and has all the options for controlling the Timeline, e.g. play, stop and so on.

Using these presets is a little bit more tricky, since you also have to determine the target for the action. That is, because you can use the playback actions not only for the current timeline, but control the playback of symbols as well. Let's have a look at how that works:

User Interaction

Assuming we have an element on the stage, with which the user can start the playback of the main composition (autoplay should therefore be deactivated!). We use Open Actions for this element (curly braces or right click) and select an appropriate event, e.g. click. Now the code view opens up and we can select the "Playback" options and opt for Play as appropriate action.

When you take a closer look at the code, you will recognize that it is yet not completed and the code is not implemented. Hence it appears in grey and between the actual code line enumeration. You are asked to confirm with "Enter", but the script still won't work yet. This is unfortunately a bit misleading. You have to set a target first, in this case, the stage, in the Pick a target panel on the right hand side.

Notice that the preset in the code view changes as soon as you make a selection: The initial "target-object" is only a place holder and therefore removed (the initial sym already refers to the stage object).

Now you have to confirm the selection by pressing "Enter" or using a double click on the target. As a result, the code actually gets inserted and is stored for this event.

Trigger

All the cases above were responding to user events. But we have already learned that there are other events as well for which we can apply actions, especially for controlling the timeline. This is where Trigger play an important role: They are set on the timeline to store an action that is then executed when the Playhead reaches this point. Very useful, indeed.

The basic principle here is the same as before, only that you don't need an extra event, the trigger iself serves as event. You can enter any (valid JavaScript) code you want here, but most likely you are opting for playback actions, for instance Play Reverse (reversing the animation) or Play From a label or certain point in time. The latter is for example useful for creating loops.

In order to create a loop, you only have to set a trigger at the end position of your animation and then select Play From. As a target you have to pick the same timeline, e.g. the stage. By default, the timer is set to 1000 milliseconds. Type in the correct time in milliseconds for the starting point of your loop, e.g. 0 to start from the very beginning.

Actions for Elements and Symbols

Actions can not only refer to the main timeline but can target elements and symbols as well. Actions can also target the timeline within a symbol in order to control its playback. The target option depends on the nature of the action. Elements for example can be the target of the Hide / Show actions, but they can not be targeted for playback actions.

Action applied to an Element

If you want to apply an action to an Element, you have look for that element in the Target panel. Note that it is subject to the context, meaning the stage or the containing symbol. That means you first have to select the appropriate context and then the element itself:

You again have to confirm the selection first before it is really applied to the event. Note that you don't have to select a subordinate element here. If you just select the container (in this case the stage) and then confirm, the action applies to this element, which allows you for example to hide and show entire symbols.

Action applied to Symbol Timeline

With the Playback Actions, you can control the timeline of any symbol that is placed in the composition. Select the appropriate action, e.g. Play or Stop for any event and then pick the according symbol as the target. You can even target a symbol within a symbol and so on. Use the Target panel to navigate to the destination. Let's say you have a trigger on the main timeline and want it to trigger the playback of a symbol (innerSymbol), that is nested inside of another symbol (outerSymbol) on the stage.

In order to get to the target you first have to select the Stage, then the outer symbol and finally the targeted inner symbol. When you take a look at the code preset here, you can find the same kind of navigation: sym is the current symbol, i.e. the Stage, then the outer symbol gets referenced, afterwards the inner symbol and finally the action is called. Finding errors

Error Messages, Debugging and Logging

In this section we will discuss the topic of potentiall errors and how to deal with them. Working with code and in a web environment has a high potential of things going wrong. JavaScript in particular is a language that is very likely to cause errors and doesn't do much to help avoiding them. We want to introduce you to the error notifications and warnings within Edge Animate itself. You will also get a quick preview on a more advanced error handling with debugging and logging.

Notifications in Edge Animate

Even when you stick to using the code wizzard for adding actions in Edge Animate, it is still likely that sooner or later, something in the code is not working properly. This might be a missing hyphen, a misplaced semicolon or a spelling mistake. This will result in code that can not be executed and therefore produce an error.

When this kind of error occurs, Edge Animate will notify you with a red error sign at the bottom right corner of the stage. By clicking on the error sign, you will get information on where the error is located and what caused the error.

The arrow at the end of the error notification will take you right to the place in your code where the error occured. It will therefore open up the code window and indicate the according line with a red dot. Also a notification at the bottom of the code window will provide more information on what has caused the error.

Don't get your hopes up to high, though. This error messaging is not suggesting solutions and you will have to look for the exact mistake yourself. Also note that only syntax errors are indicated. If you use a wrong reference or misspell the name of a variable or something like that, there will be no error message, since the code is technically not wrong.

Apart from this red error warning, Edge Animate also has a Warning notification, which indicates potential problems with browser compatibility. This is for example the case when you decide to use Filters. In case of a warning, a yellow warning sign appears at the bottom right corner of the stage. Again, a click on the sign will provide you with further information. Note that a warning means that your project will still work, whereas an error means that the corresponding code will not be executed.

The console in the browser

As we have just learned, the error notifications in Edge Animate are not helping in all the cases and there are more advanced ways to make sure your code will work properly. One of the basic assets here is the console of the browser, which can list information and notifications. Apparently, Edge Animate doesn't feature console notifications out of the box. You can learn more about this topic and how to use logging and debugging in the advanced edition of this publication.
